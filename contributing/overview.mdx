The Bad UI catalog exists to give developers a concrete, research-backed reference for the UI mistakes that silently cost products users, conversions, and trust. Every entry documents a real failure mode with statistics, accessibility citations, and a working code fix.

Contributions keep the catalog growing beyond its current 120+ documented antipatterns across 12 categories.

## Why contributions matter

The catalog is only as useful as it is complete. Many of the worst UI antipatterns are category-specific or platform-specific — a pattern that destroys mobile conversion may be invisible on desktop testing. Practitioners across different domains, frameworks, and user bases see failure modes that a small core team cannot.

Every accepted entry:

- Adds to a shared reference used by 100k+ developers
- Is licensed under CC BY 4.0 and permanently attributed
- Is linked from the relevant category page and the searchable library
- Gets peer-reviewed for accuracy before publication

## What makes a good antipattern entry

A good entry documents a **specific, observable failure** — not a vague best practice. The difference:

| Weak (not accepted) | Strong (accepted) |
|---|---|
| "Don't use low contrast" | "#9CA3AF text on #F3F4F6 background — ratio 1.9:1, WCAG minimum is 4.5:1" |
| "Make modals closeable" | "White × on #E0E0E0 background — contrast ratio 1.3:1, 99% of users fail to close on first attempt" |
| "Don't confuse users with buttons" | "Bright primary-style Cancel button next to muted Delete button — 23% accidental deletion rate" |

Every entry must have:

- A **specific, measurable description** of the bad pattern (exact colors, sizes, timings, or error rates where available)
- A **clear good alternative** with the same specificity
- At least one **WCAG 2.1/2.2 criterion** or **Nielsen heuristic** it violates
- A **user story** in the first person (a real-sounding quote from an affected user)
- A **business impact** statement with quantified consequences where possible
- A **working code fix** in React with Tailwind CSS classes

## The quality bar

<Note>
Entries are reviewed by the Bad UI editorial board before publication. The review checks for specificity, citation accuracy, code correctness, and tone. Vague or anecdotal submissions will be returned for revision.
</Note>

The catalog deliberately maintains a high editorial bar. It covers:

- **WCAG 2.1 & 2.2 compliance analysis** — every entry maps to at least one success criterion
- **Nielsen's 10 usability heuristics** — violations are cited by heuristic number and name
- **Baymard Institute research** — quantitative benchmarks for forms, checkout, and navigation patterns
- **Production-ready code fixes** — all code examples use React 19 and Tailwind CSS 4

<Warning>
Do not submit entries that duplicate existing slugs. Check the [antipattern library](https://badui.dev/library) and search by title before submitting. Duplicate slugs will be rejected at the database level by a unique constraint.
</Warning>

## Contribution process

<Steps>
  <Step title="Find or confirm the antipattern">
    Search the [library](https://badui.dev/library) to confirm the pattern is not already documented. Identify the category, gather any statistics or research citations you have, and note the specific WCAG criterion or Nielsen heuristic it violates.
  </Step>
  <Step title="Prepare all required fields">
    Before submitting, prepare every field described in the [data schema](/contributing/data-schema). Pay particular attention to `stat_line`, `bad_description`, `good_description`, `user_story`, and `code_fix` — these are the fields most often returned for revision.
  </Step>
  <Step title="Submit via the web form or SQL function">
    Use the submission form on the site at [badui.dev/submit](https://badui.dev/submit), or call the `upsert_antipattern()` SQL function directly if you have database access. Full parameter documentation is in the [submission guide](/contributing/submit-antipattern).
  </Step>
  <Step title="Review and publication">
    The editorial board reviews the entry for specificity, citation accuracy, and code correctness. Accepted entries are published with `published = true` and become visible in the catalog immediately. You will be notified by the channel you used to submit.
  </Step>
</Steps>
