Loading antipatterns occur at the points in an interface where the system and the user are out of sync. They generate uncertainty, data loss, and abandonment — particularly on slow connections where loading states are more likely to occur and more likely to be noticed.

## What this category covers

Loading antipatterns concentrate around four failure modes:

1. **Silent timeouts** — operations that fail after a delay with no indication that anything went wrong
2. **Splash screen abuse** — mandatory loading screens that delay the perceived start of the experience without providing value
3. **Missing state persistence** — user work that is lost after a page refresh, session timeout, or network error
4. **Spinnerless loading** — operations that run with no visual indication of activity

## Documented antipatterns

### Silent timeouts

A network request times out after 30 seconds. No error message appears. The loading spinner continues to spin indefinitely. The user waits, then navigates away, assuming the page is broken.

**Why it fails:** WCAG 2.2.1 (Timing Adjustable) requires that time limits have user-adjustable or extendable equivalents. More broadly, Nielsen's Heuristic #1 (Visibility of system status) requires that the system always communicate what is happening. An infinite spinner communicating nothing is a systematic failure.

**The fix:**

```tsx
const TIMEOUT_MS = 10_000;

async function fetchWithTimeout(url: string) {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), TIMEOUT_MS);

  try {
    const res = await fetch(url, { signal: controller.signal });
    clearTimeout(timer);
    return res;
  } catch (err) {
    if (err instanceof Error && err.name === "AbortError") {
      throw new Error("Request timed out. Please check your connection and try again.");
    }
    throw err;
  }
}
```

Always provide a retry mechanism alongside the timeout error message.

### 5-second mandatory splash screens

An animated logo plays for 5 seconds before the application renders. The animation is not skippable. The application was ready to render in 800 ms. The splash screen exists for brand reasons, not technical ones.

**Why it fails:** Every second a user waits for content they cannot interact with increases abandonment probability. Google's research shows that 53% of mobile users abandon sites that take over 3 seconds to load. A deliberate 5-second delay is an intentional abandonment trigger.

**The fix:** If you need a brand moment, make it the first frame of a fast-loading experience, not a gate. Use skeleton screens that immediately render structural placeholders.

```tsx
// Skeleton UI — show structure immediately
function PostSkeleton() {
  return (
    <div className="animate-pulse space-y-3">
      <div className="h-6 bg-gray-200 rounded w-3/4" />
      <div className="h-4 bg-gray-200 rounded" />
      <div className="h-4 bg-gray-200 rounded w-5/6" />
    </div>
  );
}
```

### No state persistence after refresh

A user fills out a long multi-step form, their browser crashes or the network drops, and on return all their inputs are gone. The form has no draft-saving mechanism. The user must start over.

**Why it fails:** Users experience accidental navigation (a browser crash, a mis-tap on the back button) as a regular part of using the web. Applications that destroy in-progress work without warning violate the user's reasonable expectation that the system is on their side.

**The fix:**

```tsx
// Persist form state to localStorage as the user types
const [formData, setFormData] = useState(() => {
  const saved = localStorage.getItem("form-draft");
  return saved ? JSON.parse(saved) : defaultState;
});

useEffect(() => {
  localStorage.setItem("form-draft", JSON.stringify(formData));
}, [formData]);

// Clear on successful submission
const handleSubmit = async () => {
  await submit(formData);
  localStorage.removeItem("form-draft");
};
```

## Why these patterns are harmful

| Effect | Evidence |
|---|---|
| Abandonment | 53% of mobile users abandon after 3 seconds (Google/SOASTA research) |
| Data loss | Unsaved form state means real work is lost — users rarely restart after data loss |
| Trust erosion | Infinite spinners signal a broken product, not a slow one |
| Support load | Silent timeouts generate support tickets: "The page stopped working" |

## WCAG criteria violated

| Criterion | Level | Description |
|---|---|---|
| **2.2.1** Timing Adjustable | A | Time limits must be adjustable unless they are essential |
| **2.2.5** Re-authenticating | AAA | After re-authentication, users must be able to continue without loss of data |
| **4.1.3** Status Messages | AA | Loading status must be programmatically communicated |

## Nielsen heuristics violated

- **#1 — Visibility of system status:** The system must always keep users informed about what is happening — including that a timeout has occurred.
- **#5 — Error prevention:** Preventing state loss through draft-saving is error prevention at the architectural level.
- **#9 — Help users recognise, diagnose, and fix errors:** A silent timeout gives the user nothing to recognise, diagnose, or fix.

## How to identify these patterns

<AccordionGroup>
  <Accordion title="Timeout audit">
    Throttle your network to 2G in DevTools. Trigger every async operation in the product. Note which ones: (a) show a loading indicator, (b) time out with a user-visible error, and (c) time out silently. Category (c) is a defect.
  </Accordion>
  <Accordion title="Splash screen audit">
    Measure the time between the page becoming technically interactive and the moment the user can first interact with meaningful content. Any gap over 1 second caused by intentional animation or delay is worth questioning.
  </Accordion>
  <Accordion title="State persistence audit">
    Fill out every form in the product to its halfway point. Press Cmd+R or F5. Verify whether your inputs are preserved. If they are not, there is a data loss risk.
  </Accordion>
</AccordionGroup>

<Tip>
  Skeleton screens are almost always better than spinners. They show the user the structure of what is loading, set accurate expectations about content layout, and make the load feel faster even when the actual time is identical.
</Tip>

<Note>
  Next.js App Router's `loading.tsx` file convention makes skeleton loading patterns easy to implement at the route level. Use it for every route that fetches data.
</Note>
