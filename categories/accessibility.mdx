Accessibility antipatterns exclude users with disabilities entirely, and often make interfaces harder for everyone. These failures compound: an interface with insufficient contrast, no keyboard support, and missing ARIA labels is unusable for a substantial portion of the population — and increasingly a source of legal liability.

## What this category covers

Accessibility antipatterns concentrate around four failure domains:

1. **Visual perception** — contrast ratios below WCAG minimums, content that disappears under zoom, animations that cannot be paused
2. **Keyboard operability** — controls that require a mouse, focus traps, illogical tab order
3. **Semantic structure** — missing or incorrect ARIA roles, labels, and states
4. **Cognitive accessibility** — inconsistent behavior, unclear error messages, time limits without warning

## Documented antipatterns

### Insufficient text contrast — High impact

`#9CA3AF` text on a `#F3F4F6` background produces a contrast ratio of 1.9:1. WCAG 2.1 Success Criterion 1.4.3 requires a minimum of 4.5:1 for normal text and 3:1 for large text. This combination is effectively invisible for users with low vision, color deficiencies, or standard displays in sunlit environments.

**Stat:** 45% of users over age 50 cannot read subtext at standard brightness under these color values.

**Why it fails:** Low contrast text makes your content inaccessible to users with visual impairments, including the natural decline in vision that comes with age. "Aesthetic" is never an excuse for illegible. The 1.4.3 criterion exists precisely because designer preferences consistently underestimate how many users are affected.

**The fix:**

```tsx
// Before: 1.9:1 — fails WCAG AA
<p className="text-[#9CA3AF] bg-[#F3F4F6]">
  Supporting text
</p>

// After: 4.7:1 — passes WCAG AA
<p className="text-[#1C1917] bg-[#FAFAF7]">
  Supporting text
</p>
```

Use a contrast checker (Tota11y, Chrome DevTools Accessibility panel, or the [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)) to verify every text/background color combination before shipping.

### Focus traps

A focus trap occurs when keyboard focus enters a UI component (such as a modal, dropdown, or custom widget) and cannot be moved out using standard keyboard navigation. The user is stuck.

Note: focus traps are correct behavior *inside* modal dialogs (per the ARIA Authoring Practices Guide), but they are a defect when they occur in non-modal contexts or when the modal provides no keyboard escape (Escape key or visible close button with keyboard support).

```tsx
// Modals must be escapable
useEffect(() => {
  const handleKeyDown = (e: KeyboardEvent) => {
    if (e.key === "Escape") onClose();
  };
  document.addEventListener("keydown", handleKeyDown);
  return () => document.removeEventListener("keydown", handleKeyDown);
}, [onClose]);
```

### Missing ARIA labels on icon-only controls

Buttons with only an icon and no visible text have no accessible name unless an `aria-label` or `aria-labelledby` attribute is provided. Screen readers announce them as "button" with no indication of function.

```tsx
// Before: announced as just "button"
<button onClick={close}>
  <X className="size-5" />
</button>

// After: announced as "Close modal"
<button onClick={close} aria-label="Close modal">
  <X className="size-5" aria-hidden="true" />
</button>
```

## Why these patterns are harmful

| Effect | Evidence |
|---|---|
| Legal risk | WCAG conformance is referenced by EN 301 549 (EU), Section 508 (US), and the European Accessibility Act |
| User exclusion | ~15% of the global population lives with a disability affecting their digital experience |
| Aging users | Contrast failures affect a disproportionate share of users over 50, who control significant purchasing power |
| SEO impact | Many accessibility improvements (semantic HTML, descriptive labels) also improve search indexing |

## WCAG criteria violated

| Criterion | Level | Description |
|---|---|---|
| **1.4.3** Contrast (Minimum) | AA | Normal text ≥ 4.5:1; large text ≥ 3:1 |
| **1.4.6** Contrast (Enhanced) | AAA | Normal text ≥ 7:1; large text ≥ 4.5:1 |
| **2.1.1** Keyboard | A | All functionality operable by keyboard |
| **2.4.7** Focus Visible | AA | Keyboard focus indicator must be visible |
| **4.1.2** Name, Role, Value | A | UI components must have accessible names, roles, and states |
| **1.3.1** Info and Relationships | A | Structure conveyed visually must be available programmatically |

## Nielsen heuristics violated

- **#1 — Visibility of system status:** Users relying on assistive technology must receive equivalent status information as sighted users.
- **#6 — Recognition rather than recall:** Controls must be labeled so users can identify their function without prior knowledge.
- **#8 — Aesthetic and minimalist design:** Sacrificing contrast for aesthetics is a category error — visual design must serve communication, not hinder it.

## How to identify these patterns

<AccordionGroup>
  <Accordion title="Automated testing">
    Run axe-core, Lighthouse accessibility audit, or `mint a11y` in your docs project. Automated tools catch approximately 30–40% of WCAG failures, including most contrast issues and missing ARIA attributes.
  </Accordion>
  <Accordion title="Manual keyboard audit">
    Unplug or disable your mouse. Attempt to complete every primary user flow using only the keyboard. Note every point where you become stuck or where focus becomes invisible.
  </Accordion>
  <Accordion title="Screen reader testing">
    Test with VoiceOver (macOS/iOS), NVDA (Windows), or TalkBack (Android). Verify that every interactive control has a meaningful accessible name and that state changes are announced.
  </Accordion>
  <Accordion title="Contrast checking">
    Use the Chrome DevTools Accessibility panel to inspect computed contrast ratios on any element. The panel reports the exact ratio and whether it passes AA or AAA.
  </Accordion>
</AccordionGroup>

<Warning>
  A 1.9:1 contrast ratio is not a minor issue. It fails WCAG Level A in some contexts and Level AA universally. Ship with contrast ratios this low and you are actively excluding a significant fraction of your user base.
</Warning>

<Tip>
  Install the Tota11y bookmarklet or the axe DevTools browser extension. Run both on every page you ship. Catching contrast failures at design time costs minutes; catching them post-launch costs sprints.
</Tip>
